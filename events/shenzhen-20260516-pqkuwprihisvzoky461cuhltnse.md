### 方法 3：PRD 相关要点

- **核心意思**：PRD 重要性：需求文档是项目成功的关键，需详细明确功能、实现核心点、接口请求参数、key 等信息。例如做本地微信聊天记录查看器，关键在于获取聊天记录的方式及如何对接三方工具。提前准备好相关工作，包括测试接口、准备提示词、验证表结构等，可提高项目成功率；PRD 撰写方法：可纯人肉聊，也可用 Superpower 等工具辅助，但需有评判标准，当对项目掌控度达 70% - 80% 或一半以上时，可纯人肉聊。PRD 无固定结构，可让 AI 提供模板；原型图作用：对于复杂项目，可先做原型图，如用 HTMLCSS 加 script 做小型系统，虽无实际功能但可看到交互，调整原型比调整实际代码成本低，能让开发者对系统有更清晰的认知
- **你可以怎么用**：把需求写成 PRD，用 AI 做第一版，再用截图、报错和用户反馈逼近可用版本。

  - **原型图作用**：对于复杂项目，可先做原型图，如用 HTMLCSS 加 script 做小型系统，虽无实际功能但可看到交互，调整原型比调整实际代码成本低，能让开发者对系统有更清晰的认知。
- **项目操作要点**

  - **新项目启动**：新项目需做好初始化工作，如配置环境变量、连接数据库、设置图片存储等。可利用工具生成项目相关内容，如 CF 模板，操作流程包括克隆模板、创建 TOKEN、配置 ENV 密钥、绑定域名等，但支付功能较难实现。
  - **项目迭代**：项目迭代可使用明细模式调用之前的需求文档。若基于开源项目维护，可进入 mixing 模式，让工具读取 README 总结项目信息。项目运行中可打断，但不建议，尽量批量提问题，推荐让工具跑完后再处理问题。
  - **手机端使用**：工具手机端可使用 SPEC 模式，支持将电脑作为服务端（BYOM），但需付费订阅。可在手机上编写 PRD，回到电脑上让工具运行 M 生成模式，知识库可同步，方便在手机上与工具交流想法。
- **自由交流环节**

  - **项目成本与成功率**：项目成本与 TOKEN 消耗有关，不同工具和套餐费用不同。使用 Factory Droid 工具做项目，一次成功率可达七八成。例如之前做的 schedule manager 项目，功能基本符合预期，修改一些细节后效果良好，但 TOKEN 消耗较大。
  - **需求文档优化**：有人认为需求文档编写步骤可优化以提高效率，但也有人强调需求文档是不可逆操作，需尽量详细，确保商业化和技术路线可行后再开工，避免开工后因需求不明确导致资源浪费。
  - **微信聊天记录工具**：介绍了一款抓微信聊天记录的工具，该工具从电脑内存中获取密钥解密本地 SQLite 数据库，不涉及协议，相对不易封号，但不是实时监控，只能读取内容，不能发消息。
- **后续工作计划**

  - **项目实施**：根据会议讨论的工具和方法，开展具体项目开发，按照需求文档详细规划，合理利用工具的不同模式和功能，确保项目顺利进行。
  - **成本控制**：关注 TOKEN 消耗，选择合适的套餐和反代方式，控制项目成本。
  - **经验总结**：在项目实施过程中，及时总结经验，优化需求文档编写和项目操作流程，提高工作效率和项目成功率。

# Key decisions

- **关键决策**：推荐使用 factory Droid 工具进行项目开发。

  - **问题**：寻找一款能高效完成项目开发、降低成本且功能丰富的工具。
  - **讨论方案**：
  
    - 说话人1提出 factory Droid 工具，介绍其特点和优势，如支持国产大模型、界面友好、能自动拆任务、有测试功能等。
    - 说话人对比了 factory Droid 与 CC、Codex 等工具，指出 factory Droid 在交互体验、测试功能、模型设置等方面的优势。
  - **决策依据**：factory Droid 工具具有多方面优势，能提高开发效率、保证代码质量、降低成本，且能满足不同场景的需求。

---

- **其他决策**：

  1. 对于项目开发，建议先详细准备 PRD 文档，明确功能需求和实现关键点。
  2. 可使用 HTML 做原型图，以更好地呈现项目交互和外观。
  3. 项目迭代时，可在原文件夹基础上进行，通过明细模式调用之前的需求文档。
  4. 推荐在项目跑通后再批量提出问题进行修改，避免中途打断影响效率。
  5. 对于 BYOK 模式，可注册后使用自己的 API，功能不受限制；BYOM 模式需订阅付费，可实现手机控制电脑写项目。
